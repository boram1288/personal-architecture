## 1. 결정 범위와 전제

### 1.1 설계질문 (05 문서 DP-03)

- HW IP 중재자는 Host 커널 드라이버에 둘 것인가, 별도 서비스 pVM에 둘 것인가?
- pVM 사용 구간의 SMMU/IOMMU 설정은 누가 제어하고, Host 조작을 어떻게 차단할 것인가?
- 사용 주체 전환 시 HW IP 내부 버퍼·레지스터의 잔류 데이터는 언제, 누가, 어떻게 소거할 것인가?
- 중재·전환 오버헤드는 어떤 시나리오와 수치 기준으로 측정할 것인가?
- DP-02의 어떤 최소 기능이 확보되면 HW IP 중재 PoC를 병행 착수할 수 있는가?

### 1.2 선행 DP 결정이 주는 전제

- **DP-01**: 중재자를 Host 커널에 둘 경우 스케줄링 역할만 갖는다. 격리(SMMU 설정·잔류 소거 보증)는 TCB 요소가 보장해야 한다.
- **DP-12 G3 (사활적 입력)**: SMMU 매핑 권한의 집행 주체를 본 DP에서 확정해야 한다. SMMU 설정이 비신뢰 Host 드라이버 단독에 남는 구조는 정책 모델과 충돌하므로 불허된다.
- **DP-12 G4**: 권한 철회 시 HW IP 할당 회수 경로를 본 DP 설계에 포함해야 한다.
- **DP-02 (병행)**: H1(생성)·H2(메모리)·H6(SMMU 의미론) 계약이 확정되는 즉시 PoC를 병행 착수한다.

## 2. 후보 구조 (생성: Claude)

### 후보 A: Host 커널 중재(스케줄링) + EL2 SMMU 게이트(격리)

중재자(사용 순서·시점 결정)는 Host 커널 드라이버에 두되, 격리에 해당하는 동작 — SMMU 스트림 바인딩 전환, 사용 구간 봉인, 잔류 소거 확인 — 은 EL2 기 제공 hypercall(H6)을 통해서만 수행되도록 분리한다. Host는 "다음 사용자는 pVM-X"라고 요청할 수 있을 뿐, pVM 사용 구간에 SMMU를 임의 재설정하거나 소거 단계를 건너뛸 수 없다.

- **전환 시퀀스**: (1) Host 중재자가 전환 요청 → (2) EL2가 현 사용 구간 종료 처리: HW IP 정지 확인, IP 리셋·버퍼 소거 트리거, 소거 완료 확인 → (3) EL2가 SMMU 스트림을 새 사용 주체의 Stage-2 도메인으로 재바인딩 → (4) 새 사용자 사용 개시. 소거 미완료 시 재바인딩 거부(fail-closed).
- **장점**: 신뢰 모델 전제와 정확히 일치(중재=비신뢰 스케줄링, 격리=TCB). 중재 정책(우선순위, 점유 시간)은 Host에서 자유롭게 개선 가능. 추가 TCB 없음.
- **단점**: H6(SMMU 바인딩·소거 보증 hypercall)이 기 제공 범위인지가 성립 조건(HV 확인). 소거를 EL2가 "확인"하는 방법이 HW IP별 특성(리셋 시간, 소거 수단)에 의존한다.

### 후보 B: 중재 서비스 pVM (드라이버·중재 통합 배치)

HW IP 드라이버와 중재자를 전용 서비스 pVM에 두고, Host와 보안 pVM 모두 이 pVM에 사용을 요청한다.

- **장점**: 중재자가 TCB이므로 중재 결정 자체를 신뢰할 수 있고, SMMU 제어를 서비스 pVM에 위임하는 구조도 가능하다.
- **단점**: ISP·NPU 드라이버 전체를 pVM으로 포팅해야 하고(개발량·일정 리스크), Host 일반 기능의 모든 HW IP 사용이 pVM 왕복을 거쳐 상시 오버헤드가 생긴다(QA-11 역행). 서비스 pVM TCB 편입 조건(DP-01)과 부트스트랩, 서비스 pVM 상한 정책(DP-11 결정)에도 부담이 된다.

### 후보 C: 정적 모드 전환 (시분할 파티셔닝)

동적 중재 대신, "보안 파이프라인 활성 구간"과 "Host 일반 사용 구간"을 명시적 모드로 나누고 모드 전환 시에만 소유권을 바꾼다.

- **장점**: 전환 빈도가 낮아 잔류 소거·재바인딩 검증이 단순하고 PoC가 가장 빠르다. 검증·증빙(QA-10)이 쉽다.
- **단점**: Host 일반 촬영과 보안 파이프라인의 동시 사용(VOS-02, CONST-05의 "공유 자원" 요구)을 충족하지 못한다 — 프레임 단위 교차 사용이 불가능해 R-2를 부분 충족에 그친다. 최종 구조로는 부적합하고 PoC 1단계로만 의미가 있다.

### 공통 설계 요소

- **오버헤드 측정 시나리오**: (S1) Host 1080p30 일반 촬영 + Secure Camera 1080p30 동시 — 프레임 교차 전환 빈도에서 양측 프레임율 저하 10% 이내(QS-11), (S2) 전환 1회 비용(정지 확인+소거+재바인딩) 1ms 이하 목표(가정치), (S3) NPU 추론 배치 교차 시나리오.
- **DP-02 병행 착수 조건**: H1·H2 확정 + H6 의미론 합의(시그니처 수준) 시점에 PoC 착수. PoC 1단계는 후보 C 방식(저빈도 전환)으로 격리·소거를 먼저 검증하고, 2단계에서 전환 빈도를 올려 후보 A의 동시성 목표를 검증한다.

### 평가 기준

| 기준 | 설명 |
|------|------|
| E1. DMA 격리 보장 | pVM 사용 구간의 SMMU 제어가 비신뢰 Host 단독에 있지 않은가 (QA-03, DP-12 G3) |
| E2. 잔류 소거 보증 | 전환 시 소거가 우회 불가능하게 강제되는가, 소거 확인 주체 (QA-03) |
| E3. 전환 오버헤드 | 동시 사용 시나리오에서 QA-11 목표(전환 1ms, 저하 10% 이내) 달성 가능성 |
| E4. 제약 적합 | CONST-02(EL2 수정 불가 — H6 기 제공 여부), CONST-05(HW 변경 불가) |
| E5. Host 일반 기능 영향 | Host 단독 사용 구간의 성능·드라이버 변경 범위 (VOS-02) |
| E6. 구현·PoC 리스크 | 드라이버 포팅량, 일정 내 PoC 가능성, 단계적 검증 경로 |
